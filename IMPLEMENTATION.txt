1. EXECUTIVE SUMMARY
================================================================================

This application is an outbound voice AI agent for healthcare-style appointment
reminders. A clinic operator uses a Streamlit UI to:

  - Create appointments in MongoDB
  - Enter a phone number (E.164 format, e.g. +15551234567)
  - Choose a voice provider and start an outbound call

The agent persona is "Alex" from HealthCare Plus Clinic. The call goal is to:

  - Confirm patient identity
  - Remind them of date, time, provider, and location
  - Obtain confirmation, reschedule request, or cancellation
  - Persist transcript and outcome in MongoDB

TWO IMPLEMENTATION PATHS are supported:

  A) CUSTOM PIPELINE ("provider": "custom")
     You own every layer: Twilio telephony, Deepgram STT, Groq LLM, Edge TTS.
     Best for demonstrating architecture and full control.

  B) VAPI PIPELINE ("provider": "vapi")
     Vapi runs the real-time voice stack; your FastAPI app triggers calls via
     REST and receives end-of-call webhooks. Best for faster production setup
     while keeping the same scenario prompts and MongoDB logging.

Both paths share: FastAPI backend, MongoDB, Streamlit UI, appointment scenario.

3. DUAL VOICE PROVIDER DESIGN
================================================================================

Why two providers?
  - Assignment asks for a self-built STT/LLM/TTS pipeline (custom).
  - Vapi is added as a production-ready alternative with less moving parts.
  - Same business logic (scenario, context, outcomes) applies to both.

Selection:
  - API: POST /api/calls/outbound  body: { "provider": "custom" | "vapi", ... }
  - UI:  Radio button "Voice provider" on Place call tab

Provider comparison:

  | Aspect          | custom                    | vapi                          |
  |-----------------|---------------------------|-------------------------------|
  | Telephony       | Your Twilio number        | Vapi phoneNumberId            |
  | STT             | Deepgram (your key)       | Deepgram (via Vapi assistant) |
  | LLM             | Groq (your key)           | Groq (via Vapi assistant)     |
  | TTS             | Edge TTS (local)          | OpenAI/11labs/etc (Vapi)      |
  | Real-time logic | app/services/media_stream | Vapi cloud                    |
  | Transcript      | Live append to MongoDB    | end-of-call-report webhook    |
  | Needs ngrok     | Yes (webhooks + WS)       | Yes (webhook URL)             |
  | Needs FFmpeg    | Yes (MP3 -> mulaw)        | No                            |

5. DATA MODEL (MONGODB)
================================================================================

Database name: voicebot (config: MONGODB_DB_NAME)

Collection: appointments
  - patient_name: string
  - patient_phone: string (E.164)
  - appointment_datetime: datetime
  - provider_name: string
  - clinic_name: string
  - clinic_address: string
  - notes: string (optional)
  - status: scheduled | confirmed | reschedule_requested | cancelled
  - created_at, updated_at: datetime

Collection: calls
  - phone_number: string
  - scenario: string (default appointment_reminder)
  - provider: custom | vapi
  - appointment_id: ObjectId reference (optional)
  - context: object (snapshot used during call)
  - status: queued | ringing | in_progress | completed | failed | no_answer
  - twilio_call_sid: string (custom path)
  - vapi_call_id: string (vapi path)
  - conversation: array of { role, content, timestamp, metrics? }
    metrics (assistant turns): { llm_ms, tts_ms, total_ms, target_ms, within_target }
  - outcome: string (confirmed, cancelled, reschedule_requested, etc.)
  - created_at, updated_at: datetime

================================================================================

File: app/scenarios/appointment_reminder.py

Persona: Alex — warm, professional scheduling assistant.

System prompt rules:
  - Short spoken responses (1-3 sentences)
  - Confirm identity before appointment details

7. CUSTOM PIPELINE (TWILIO + DEEPGRAM + GROQ + EDGE TTS)
================================================================================

7.1 Outbound dial
  - POST /api/calls/outbound with provider=custom
  - CallRepository creates call doc
  - TwilioService.place_outbound_call() uses Twilio REST API
  - TwiML URL: {PUBLIC_BASE_URL}/webhooks/twilio/voice?call_id=...

7.2 TwiML webhook (webhooks.py)
  Returns XML:
    <Connect><Stream url="wss://.../ws/media">
      <Parameter name="call_id" value="..." />
    </Stream></Connect>

7.3 Media Stream session (media_stream.py)
  On "start" event:
    - Load call + appointment context from MongoDB
    - Speak opening_line via Edge TTS

  On "media" events:
    - Forward mulaw audio to Deepgram WebSocket (nova-2, 8kHz)

  On Deepgram final transcript:
    - GroqConversationService.generate_reply()
    - synthesize_mulaw() + chunk to Twilio

  On outcome keywords:
    - Update appointment status in MongoDB

7.4 Audio format
  - Twilio sends/receives base64 mulaw 8kHz mono
  - Edge TTS outputs MP3 -> pydub + FFmpeg -> PCM -> audioop.lin2ulaw

7.5 Production voice-bot rules (custom pipeline)

  PIPELINE ORDER (never reorder):
    Twilio -> Deepgram (STT) -> Groq (LLM) -> Edge TTS -> Twilio

  STT — final transcripts only (app/services/deepgram_stt.py):
    - interim_results=false — never send partial transcripts to Groq
    - vad_events=true — SpeechStarted triggers barge-in
    - Only is_final Results trigger the LLM pipeline

  LLM — voice-optimized prompts (appointment_reminder.py):
    - Short system prompt, 1-2 sentence replies
    - No markdown, bullets, or symbols (TTS reads them aloud)
    - Groq max_tokens=100 for lower latency

  Interruption / barge-in (media_stream.py):
    1. Twilio "clear" event stops buffered TTS playback
    2. Cancel in-flight asyncio pipeline task (Groq request)
    3. Conversation history is preserved (not wiped)
    4. Deepgram WebSocket reconnects fresh for next utterance

  Async architecture:
    - Twilio recv loop never blocked by LLM/TTS
    - Each user turn runs in asyncio.create_task(_run_turn_pipeline)
    - Errors in one component do not crash the call (try/except per stage)

  Latency logging (turn_metrics.py + MongoDB conversation.metrics):
    - Logs llm_ms, tts_ms, total_ms per assistant turn
    - Target: user end-of-speech -> bot audio starts in under 1500ms
    - Check API logs: turn_latency ... within_target=True/False

  WebSocket mandatory:
    - HTTP cannot carry live bidirectional audio
    - Twilio Media Stream -> FastAPI /ws/media

7.6 Required env vars
  - TWILIO_ACCOUNT_SID / TWILLIO_* (typo supported)
  - TWILIO_AUTH_TOKEN / TWILLIO_AUTH_TOKEN
  - TWILIO_PHONE_NUMBER
  - DEEPGRAM_API_KEY or STT=
  - GROQ_API_KEY
  - PUBLIC_BASE_URL (HTTPS ngrok)

8. VAPI PIPELINE
================================================================================

8.1 Overview
  Vapi hosts the real-time voice conversation. Your backend:
    1. Creates a call record in MongoDB
    2. POST https://api.vapi.ai/call
    3. Receives webhooks at /webhooks/vapi

8.2 Transient assistant (default)
  File: app/scenarios/vapi_appointment.py

  Built per call with:
    - model.provider: groq
    - model.model: GROQ_MODEL from env
    - transcriber.provider: deepgram (nova-2)
    - voice: VAPI_VOICE_PROVIDER / VAPI_VOICE_ID (default openai/alloy)
    - firstMessage: same opening_line as custom pipeline
    - serverUrl: {PUBLIC_BASE_URL}/webhooks/vapi

  NOTE: Link Groq (and voice provider) in Vapi dashboard under Integrations.

8.3 Saved assistant (optional)
  If VAPI_ASSISTANT_ID is set, that assistant is used instead of transient.
  Configure the assistant in https://dashboard.vapi.ai to match the scenario.

8.4 Vapi phone number
  - Import Twilio number into Vapi OR use a Vapi free number
  - Set VAPI_PHONE_NUMBER_ID from dashboard

8.5 Webhooks (vapi_webhooks.py)
  Handles server messages:
    - status-update -> update call status in MongoDB
    - end-of-call-report -> save transcript, detect outcome, mark completed

  Call correlation uses metadata.call_id set when creating the outbound call.

8.6 Required env vars
  - VAPI_API_KEY
  - VAPI_PHONE_NUMBER_ID
  - PUBLIC_BASE_URL (for serverUrl)
  - GROQ_MODEL (for transient assistant)
  - Optional: VAPI_ASSISTANT_ID, VAPI_VOICE_PROVIDER, VAPI_VOICE_ID

8.7 Vapi dashboard setup
  1. Create account at https://dashboard.vapi.ai
  2. Add Groq API key under Integrations
  3. Add/import phone number -> copy phoneNumberId
  4. Either rely on transient assistant OR create assistant + set VAPI_ASSISTANT_ID
  5. Set Server URL to: https://YOUR-NGROK/webhooks/vapi
     (Also set per-assistant serverUrl if using saved assistant without transient)

15. TROUBLESHOOTING
================================================================================

API Offline in UI
  -> Run run_api.bat; check port 8000

Custom: call connects, silence
  -> FFmpeg on PATH; check TTS errors in API logs
  -> Verify DEEPGRAM_API_KEY / GROQ_API_KEY

Custom: Twilio never streams
  -> PUBLIC_BASE_URL must be HTTPS ngrok
  -> WebSocket wss:// must be reachable

Vapi: call fails immediately
  -> VAPI_API_KEY and VAPI_PHONE_NUMBER_ID set?
  -> Groq linked in Vapi dashboard?
  -> Trial number restrictions (verified destinations)

Vapi: no transcript after call
  -> serverUrl must point to /webhooks/vapi
  -> Check ngrok inspector for POST payloads

MongoDB errors
  -> MONGODB_URI correct; IP allowlist in Atlas

Wrong Python packages
  -> Always use venv: venv\Scripts\python.exe -m pip install -r requirements.txt
